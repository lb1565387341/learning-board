# 第 1 课 · String 底层

## 📖 速记卡

```
SDS 设计                                   Redis String

1. Redis 的 String 底层是 SDS（简单动态字符串）
   不是 C 语言的 char*

2. SDS 三个优势 vs C 字符串：
   ① O(1) 取长度（C 需要遍历 O(N)）
   ② 二进制安全（存 \0 不截断，C 遇 \0 就停）
   ③ 自动扩容防溢出（alloc 记录了总容量）
   ④ 空间预分配（减少 realloc 次数）

3. 三种编码类型（OBJECT ENCODING key 查看）：
   int     → 纯数字，8 字节以内，直接存 long
   embstr  → ≤44 字节，一次 malloc，连续内存
   raw     → >44 字节，两次 malloc，对象和 SDS 分开

4. 44 字节的来历：
   jemalloc 分配 64 字节块，-20（RedisObject 头）
   = 44 留给 SDS，正好零碎片

5. Java 里的乱码 \xac\xed 是 JDK 序列化头
   换成 StringRedisSerializer 解决

6. 高频命令：GET/SET/SETNX/SETEX/INCR/DECR
   SETNX 和 SET NX EX 区别：后者是原子操作
```

## ⚡ 超快记忆

```
Q：SDS 比 C 字符串改进在哪？
→ ① 长度 O(1)  ② 二进制安全（存图片/\0 不截断）  ③ 自动扩容防溢出  ④ 空间预分配（减少 realloc）

Q：String 三种编码？
→ int（纯数字，8B 以内）、embstr（≤44B 一次分配）、raw（>44B 两次分配）

Q：44 字节怎么来的？
→ jemalloc 64B - RedisObject 头 20B = 44B

Q：SETNX 和 SET NX EX 区别？
→ SETNX 非原子（需 +EXPIRE），SET NX EX 一条命令原子操作

Q：Java 取到 \xac\xed 开头是什么？
→ JDK 序列化头，换 StringRedisSerializer 解决
```

# 第 2 课 · Hash ziplist → hashtable

## 📖 速记卡

```
Hash 底层编码                            ziplist 升级条件

1. Hash 底层有两种编码（OBJECT ENCODING key 查看）：
   ziplist   → 连续内存块，省内存，O(N) 遍历
   hashtable → 标准 dict，O(1) 读写

2. ziplist 结构（来源：Redis 源码 ziplist.c）：
   <zlbytes><zltail><zllen><entry><entry>...<entry><zlend>
    4B        4B       2B                       1B

   每个 entry: <prevlen><encoding><data>

3. prevlen 编码规则：
   上一个 entry < 254B → 1 字节
   上一个 entry >= 254B → 5 字节（源码 ZIP_BIG_PREVLEN=254）

4. 升级条件（官方文档 Memory optimization）：
   hash-max-ziplist-entries 512  ← 超过 512 个 field
   hash-max-ziplist-value    64  ← 某个 value 超过 64 字节
   → 或关系，一个触发就升级，单向不降级

5. 连锁更新（源码 __ziplistCascadeUpdate）：
   delta = 4（prevlen 1B→5B 增加 4 字节）
   原 entry >= 250 → +4 >= 254 → 触发下一个连锁
   默认 value 64B 离 250 很远，实际几乎不触发

6. 为什么 512/64？
   官方说 "CPU/memory tradeoff" 和 "up to 10x less memory"
   具体值没说理由，是 Redis 工程验证的默认值
```

## ⚡ 超快记忆

```
Q：Hash 的两种编码是什么？
→ ziplist（小数据省内存 O(N)）和 hashtable（大数据 O(1) 读写）

Q：ziplist 升级 hashtable 的条件？
→ ① field > 512 个  ② 某个 value > 64 字节。或关系，单向升级不降级

Q：什么是连锁更新？临界值是多少？
→ entry 变大导致下一个 prevlen 1B→5B（多 4 字节）。原 entry ≥ 250 会触发下一个

Q：prevlen 的编码规则？
→ 上一个 entry < 254B → 1B；≥ 254B → 5B（源码 ZIP_BIG_PREVLEN=254）

Q：为什么 Redis 不担心连锁更新？
→ 默认 value 64B 离 250 很远，实际很难连续多个 entry 恰好在临界区

Q：ziplist 和 hashtable 各有什么优缺点？
→ ziplist 省内存但 O(N)；hashtable O(1) 但更费内存

Q：官方文档对 512/64 怎么说？
→ "CPU/memory tradeoff"，能节省最多 10 倍内存（平均 5 倍）
```
